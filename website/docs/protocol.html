Overview

Protocol v6 closes the constitutional gap that v5 left open. v5 described one governed repo. v6 governs both the repo-local workflow and the initiative-level coordinator that can sequence several governed repos under one super_run_id.

Authority split: repo-local accepted history remains the source of truth for accepted work inside a repo. The coordinator derives projections, barriers, pending initiative gates, and cross-repo context from that repo-local truth.

Repo-local run

The v5 governed state machine still handles actual turn assignment, staging, validation, acceptance, rejection, and repo-local human gates.

Coordinator initiative

The v6 coordinator sequences workstreams across repos, requests initiative gates, records acceptance_projection events, and writes cross-repo context artifacts.

Constitutional command

agentxchain multi step is not a dumb dispatcher. It reconciles repo-local truth first, then dispatches or requests the next coordinator gate.

Repo-Local Governed Run

Protocol v6 does not replace the repo-local governed lifecycle. It builds on it. Each participating repo still owns its own agentxchain.json, run state, dispatch bundles, staging files, and accepted history.

agentxchain.json
.agentxchain/state.json
.agentxchain/history.jsonl
.agentxchain/decision-ledger.jsonl
.agentxchain/dispatch/turns/<turn_id>/
.agentxchain/staging/<turn_id>/turn-result.json

Repo-local commands remain authoritative for actual turn completion:

  • agentxchain step
  • agentxchain accept-turn
  • agentxchain reject-turn
  • agentxchain approve-transition
  • agentxchain approve-completion

Coordinator Initiative

The coordinator lives in the workspace root and persists initiative state under .agentxchain/multirepo/. Its primary entities are super_run_id, repo_runs, workstreams, barriers, pending gates, and acceptance projections.

super_run_id

The initiative-wide identifier written at multi init time and included in coordinator history, gate requests, and hook payloads.

acceptance_projection

A derived history event that projects one repo-local accepted turn into coordinator space with projection_ref, repo_turn_id, files_changed, decisions, and verification evidence.

Barrier snapshot

Each workstream has a completion barrier recorded in barriers.json. Barrier transitions are append-only in barrier-ledger.jsonl.

The operator surface is explicit:

agentxchain multi init
agentxchain multi status
agentxchain multi step
agentxchain multi approve-gate
agentxchain multi resync

Coordinator Config

The coordinator contract is declared in agentxchain-multi.json. The current config schema value is "0.1".

{
  "schema_version": "0.1",
  "project": { "id": "agentxchain-dev", "name": "AgentXchain.dev" },
  "repos": {
    "api": { "path": "./repos/api", "default_branch": "main", "required": true },
    "web": { "path": "./repos/web", "default_branch": "main", "required": true }
  },
  "workstreams": {
    "delivery": {
      "phase": "implementation",
      "repos": ["api", "web"],
      "entry_repo": "api",
      "depends_on": [],
      "completion_barrier": "all_repos_accepted"
    }
  },
  "routing": {
    "implementation": { "entry_workstream": "delivery" }
  },
  "gates": {
    "initiative_ship": {
      "requires_human_approval": true,
      "requires_repos": ["api", "web"]
    }
  },
  "hooks": {
    "before_assignment": [],
    "after_acceptance": [],
    "before_gate": [],
    "on_escalation": []
  }
}
Field Rule
workstreams.*.phase Must be planning, implementation, or qa
workstreams.*.completion_barrier Must be all_repos_accepted, interface_alignment, ordered_repo_sequence, or shared_human_gate
workstreams.*.entry_repo Must also appear in the workstream repo list
workstreams.*.depends_on Must reference declared workstreams and remain acyclic
routing.<phase>.entry_workstream Must exist and belong to the same phase
gates.*.requires_repos Must reference declared repos only

Gates And Resync

agentxchain multi step enforces three invariants before dispatch:

  • Do not dispatch when the initiative is already completed.
  • Do not dispatch when a coordinator gate is pending; require agentxchain multi approve-gate.
  • Resync from repo-local authority before selecting the next assignment when divergence is detected.
Important: after_acceptance hooks fire only for newly projected acceptances discovered during resync, not for arbitrary coordinator churn.

Phase transition

When the current phase is satisfied, multi step requests a coordinator phase gate automatically and pauses the initiative until agentxchain multi approve-gate is run.

Initiative completion

When the final phase is satisfied and required repos are completion-ready, multi step requests the initiative completion gate automatically and the operator approves it with the same command surface.

Coordinator history records these gate events:

  • phase_transition_requested
  • phase_transition_approved
  • run_completion_requested
  • run_completed

Cross-Repo Context

Every coordinator dispatch generates a context snapshot for the target repo. The coordinator writes it under .agentxchain/multirepo/context/<context_ref>/ and copies it into the repo-local dispatch bundle as COORDINATOR_CONTEXT.json and COORDINATOR_CONTEXT.md.

The snapshot includes:

  • upstream_acceptances from already accepted upstream repos
  • active_barriers that still constrain the target repo
  • required_followups derived from dependencies, accepted outputs, and barrier notes

The coordinator also appends a context_generated event to history.jsonl with context_ref, target_repo_id, relevant_workstream_ids, and upstream_repo_ids.

Invalidation is informational. When a later acceptance makes downstream context stale, the coordinator computes context_invalidations for hooks and operator tooling. The invalidation signal does not itself mutate coordinator state.

Coordinator Hooks

Coordinator hooks are declared in agentxchain-multi.json and run through four phases distinct from repo-local hooks:

Phase Type Use
before_assignment Blocking Prevent dispatch before the next repo-local turn is assigned
after_acceptance Advisory React to new acceptance projections, barrier effects, and context_invalidations
before_gate Blocking Prevent coordinator phase/completion approval while keeping pending gate state coherent
on_escalation Advisory Notify when the coordinator enters blocked state

Common payload fields include super_run_id, workstream_id, repo_id, repo_run_id, pending_barriers, pending_gate, coordinator_status, and coordinator_phase.

after_acceptance adds projection_ref, barrier_effects, and context_invalidations. before_gate adds gate metadata. on_escalation adds blocked_reason and repo_runs.

Tamper protection: coordinator hooks may not mutate coordinator-owned files under .agentxchain/multirepo/ and may not mutate protected repo-local orchestrator files or existing dispatch bundle artifacts. Protected-file tamper is rolled back and treated as a protocol violation.

Compatibility

Protocol v6 extends the v5 single-repo model; it does not erase it. Existing repo-local governed projects still use the v5 governed surface. The new coordinator layer is additive.

  • Use SPEC-GOVERNED-v5.md when you need the frozen v1.1 single-repo reference.
  • Use PROTOCOL-v6.md when you need the current multi-repo constitutional contract.
  • Use /docs/protocol-v6.html when you need a stable permalink for the current protocol version.
Governed multi-agent software delivery. The protocol is versioned, explicit, and auditable because coordination claims without a constitution are not credible.
