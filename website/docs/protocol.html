Docs / Protocol

The protocol is the product. This is the protocol.

AgentXchain is a governed convergence protocol for AI software teams. It assigns roles, requires structured turn results, validates artifacts, enforces phase gates, and maintains append-only history. The runner is replaceable. The protocol is the durable layer.

Overview

The protocol governs how multiple AI agents collaborate on a single codebase. It does not build agents, connect them to tools, or decide what they work on. It governs the workflow between them: who can work, what they must produce, how results are validated, when phases advance, how conflicts are resolved, and where decisions are recorded.

Model-agnostic. A team can run Claude for planning, GPT for implementation, and Gemini for review — all governed by the same rules, all producing structured artifacts against the same state.

v1.1 scope

  • One project repo per run
  • Parallel governed turns — up to 4 concurrent (v1.1)
  • Sequential turns as default (backward compatible with v1.0)
  • Role-based routing with human checkpoints
  • Structured turn results (JSON) plus human-readable prose
  • Three runtime types: manual, local_cli, api_proxy
  • Four artifact types: workspace, patch, commit, review
  • Orchestrator-derived artifact truth (not agent self-reporting)
  • Adapter-local auto-retry for api_proxy (v1.1, opt-in)
  • Preemptive tokenization for api_proxy (v1.1, opt-in)
  • Provider-specific error mapping (v1.1, automatic)
  • Persistent blocked state with structured recovery descriptors (v1.1)

Runtime model

The orchestrator manages a run using three entities. Everything else — TALK.md, history.jsonl, decision-ledger.jsonl, phase summaries — is derivative.

Run State

The accepted state of the project and workflow. Contains run status, current phase, active turns, budget, blocked state, and pending approvals. Written only by the orchestrator.

Turn Assignment

Which role(s) are currently authorized to act. Each assignment captures a baseline (git HEAD, clean status, timestamp), a deadline, and a turn sequence number.

Turn Result

The structured output of one completed turn. Contains summary, decisions, objections, files changed, verification evidence, artifact reference, and routing suggestions.

Configuration

Static project configuration in agentxchain.json, checked into the repo. Schema version "1.0" or "1.1". A v1.0 config with no new fields runs identically under v1.1.

Top-level shape

{
  "schema_version": "1.0",
  "project": { "id": "", "name": "", "default_branch": "main" },
  "roles": {},
  "runtimes": {},
  "routing": {},
  "gates": {},
  "budget": {},
  "retention": {},
  "prompts": {},
  "rules": {}
}

Retry policy (v1.1, opt-in)

{
  "retry_policy": {
    "enabled": true,
    "max_attempts": 3,
    "base_delay_ms": 1000,
    "max_delay_ms": 8000,
    "backoff_multiplier": 2,
    "jitter": "full",
    "retry_on": ["rate_limited", "network_failure", "timeout"]
  }
}

Retries are adapter-local. They do NOT create governed turns or mutate governed attempt counters. Delay formula: min(base_delay_ms * backoff_multiplier^attempt, max_delay_ms) with optional full jitter. Non-retryable classes (invalid_request, budget_error, context_overflow) are never retried.

Preemptive tokenization (v1.1, opt-in)

{
  "context_window_tokens": 200000,
  "preflight_tokenization": {
    "enabled": true,
    "tokenizer": "provider_local",
    "safety_margin_tokens": 2048
  }
}

Token budget: context_window_tokens - max_output_tokens - safety_margin_tokens. If estimated input exceeds budget, attempt bounded compression. If still over, fail locally with context_overflow without making the API call.

Routing

Each phase declares an entry role, allowed next roles, exit gate, and max concurrent turns.

FieldDescription
entry_roleDefault role for the phase
allowed_next_rolesHard allowlist; human is reserved
exit_gateGate ID that must pass before phase advancement
max_concurrent_turnsMaximum parallel turns (v1.1). Default: 1. Cap: 4.

Gates

PredicateBehavior
requires_filesListed file paths must exist
requires_verification_passTurn verification must be pass or attested_pass
requires_human_approvalBlocks auto-advance; run pauses for approve-transition

Validation rules

  • Unknown top-level keys are rejected
  • Every role must reference an existing runtime
  • Every routing role must exist or be human
  • review_only roles cannot use local_cli runtimes
  • api_proxy runtimes may only bind to review_only roles (v1.1)
  • argv transport requires {prompt} placeholder in command
  • max_concurrent_turns must be >= 1 and <= 4

State machine

Run statuses

StatusMeaning
idleInitialized, no run started
activeRun in progress, turns may be assigned and accepted
pausedWaiting for a known human approval (phase transition or run completion)
blockedCannot make forward progress; requires human diagnosis or external action

blocked vs paused (v1.1): paused = the operator knows exactly what to do (approve-transition or approve-completion). blocked = the operator must diagnose and resolve a problem before resuming. blocked requires a structured BlockedReason with category, recovery, timestamp, and turn ID.

Key rules

  • The orchestrator never enters accepting for an unvalidated turn result
  • Phase advancement is gate-driven and orchestrator-enforced, never from agent suggestion alone
  • Phase transitions require all active turns resolved before gate evaluation (v1.1)
  • Acceptance never auto-assigns the next turn
  • Retry is per turn assignment, not per phase
  • History is append-only
  • blocked blocks new assignment but does not cancel running sibling turns (v1.1)
  • Acceptance is serialized via an acceptance lock (v1.1)

Conflict detection (v1.1)

File-level conflict detection at acceptance time. Two turns conflict when their observed changed files overlap AND the later turn was assigned before the earlier turn was accepted.

Operator resolution paths:

  • reject_and_reassign — same turn_id, increment attempt, re-baseline, redispatch with conflict context
  • human_merge — operator manually resolves files, re-runs acceptance pipeline
No silent auto-merge or auto-accept for conflicts. Repeated re-conflict is bounded by a loop guard.

Turn result contract

Every turn produces this shape. The orchestrator validates it before acceptance.

{
  "schema_version": "1.0",
  "run_id": "...",
  "turn_id": "...",
  "role": "...",
  "runtime_id": "...",
  "status": "completed | blocked | needs_human | failed",
  "summary": "...",
  "decisions": [
    { "id": "DEC-NNN", "category": "...", "statement": "...", "rationale": "..." }
  ],
  "objections": [
    { "id": "OBJ-NNN", "severity": "low|medium|high|blocking", "statement": "...", "status": "raised" }
  ],
  "files_changed": ["..."],
  "artifacts_created": ["..."],
  "verification": {
    "status": "pass | fail | skipped",
    "commands": ["..."],
    "evidence_summary": "...",
    "machine_evidence": [{ "command": "...", "exit_code": 0 }]
  },
  "artifact": { "type": "workspace|patch|commit|review", "ref": "..." },
  "proposed_next_role": "...",
  "phase_transition_request": "implementation | qa | null",
  "run_completion_request": false,
  "needs_human_reason": "... | null",
  "cost": { "input_tokens": 0, "output_tokens": 0, "usd": 0 }
}

Required invariants

  • run_id and turn_id must match current assignment
  • role and runtime_id must match current assignment
  • proposed_next_role must be routing-legal or human
  • review_only roles must not declare product file changes
  • review_only roles must have artifact.type = "review"
  • phase_transition_request and run_completion_request are mutually exclusive
  • Decision IDs match DEC-NNN, objection IDs match OBJ-NNN

Cost accounting

  • Provider telemetry is the authoritative cost source
  • Hardcoded model rates are fallback estimation only
  • When auto-retry is enabled, the success-path cost aggregates usage across all adapter-local attempts

Five stages, executed in order. Short-circuits on first failure.

Dispatch bundles

Dispatch bundles are turn-scoped. When only one turn is active, there is simply one live bundle:

.agentxchain/dispatch/turns/<turn_id>/ASSIGNMENT.json
.agentxchain/dispatch/turns/<turn_id>/PROMPT.md
.agentxchain/dispatch/turns/<turn_id>/CONTEXT.md
.agentxchain/staging/<turn_id>/turn-result.json

Parallel turn isolation (v1.1)

When max_concurrent_turns > 1, multiple turn-scoped bundles coexist. dispatch/index.json is the operator-visible manifest listing all active dispatch bundles.

Conflict context in redispatch

When a turn is redispatched after reject_and_reassign, the bundle carries structured conflict context: which files conflicted, which turn was accepted ahead, and the new baseline ref.

Dispatch bundles are ephemeral projections, re-materialized on every resume/retry.

Repo observation

The orchestrator independently verifies what happened in the repo. Agents do not self-report truth — the orchestrator observes it.

Baseline capture

At turn assignment: Git HEAD ref, working tree clean status, and timestamp.

Change observation

At acceptance: actual files changed (via git diff against baseline), current HEAD ref, and diff summary.

Declared vs observed

  • Authoritative roles: undeclared file changes are an artifact_error
  • Review-only roles: any observed product file changes are an artifact_error

Integration ref derivation

accepted_integration_ref is derived from orchestrator observation, not from the agent's artifact.ref. It is a git anchor. The exact accepted state is the combination of accepted_integration_ref and the corresponding history.jsonl row's observed_artifact.

Operational path exclusion

Orchestrator-owned paths (.agentxchain/dispatch/, .agentxchain/staging/) are excluded from actor observation and the clean baseline check.

Challenge requirement

The core protocol differentiator. Enforced mechanically, not suggested politely.

Mandatory objection: review_only roles MUST have a non-empty objections array. Empty objections from a review role is a protocol_error — the turn is rejected.
  • review_only roles: objections required, always
  • authoritative roles: may have empty objections only on the first turn in a phase
  • Objection substance is not validated by the orchestrator — but the structural requirement is enforced

This is adversarial robustness applied to software delivery. The quality of collective output depends not on the intelligence of participants, but on the structure of their disagreement.

Reserved paths

Agents must not write to these orchestrator-owned files:

  • .agentxchain/state.json
  • .agentxchain/history.jsonl
  • .agentxchain/decision-ledger.jsonl
  • .agentxchain/lock.json

v1.0 compatibility

Config compatibility

A valid v1.0 agentxchain.json with schema_version: "1.0" and no v1.1 fields runs identically under v1.1. No config changes required to upgrade.

State migration

v1.1 reads and migrates "1.0" state files in place. Key migration steps:

  • current_turn (if present) migrates to active_turns[turn_id]
  • paused + blocked_on migrates to status: "blocked" with structured blocked_reason
  • schema_version updated to "1.1"
v1.0 does NOT read "1.1" state files. There is no forward compatibility guarantee.

Behavioral preservation

With default settings (no retry_policy, no preflight_tokenization, max_concurrent_turns = 1 or absent): sequential single-turn behavior is unchanged. Turn lifecycle, validation, acceptance, and rejection are identical to v1.0.

Governed multi-agent software delivery. Agents are required to challenge each other, and humans approve the moments that define the outcome.
