Docs / CLI

The governed command surface, without the hand-waving.

These commands are the operator contract for governed runs. The important distinctions are structural: resume assigns or redispatches work without waiting, step runs one governed turn lifecycle, and approval commands are how humans keep constitutional authority over phase changes and ship decisions.

The two commands people confuse

resume

Assignment-only. It starts or resumes a run, assigns a turn, or redispatches a retained turn, then exits. Use it when you want the orchestrator to set up work without waiting for the runtime to finish.

step

Full turn execution. It initializes or resumes the run, assigns or targets one turn, dispatches it, waits for a staged result, validates it, and accepts or rejects it. One invocation handles one turn lifecycle only.

Human approval commands

approve-transition and approve-completion are not optional polish. They are the constitutional gates that keep agents from advancing phases or self-shipping.

Real governed artifact paths: dispatch bundles live under .agentxchain/dispatch/turns/<turn_id>/, staged results live under .agentxchain/staging/<turn_id>/turn-result.json, and accepted evidence is appended to .agentxchain/history.jsonl and .agentxchain/decision-ledger.jsonl.

Command map

Turn lifecycle commands

agentxchain init --governed [--template <id>] [--yes]

Creates the governed scaffold only. It does not start a run. The initial project remains idle in the planning phase until resume or step initializes a run and assigns a turn.

Use --template when you want scaffold intent recorded explicitly. Built-ins are generic, api-service, cli-tool, and web-app. Non-generic templates add project-type-specific planning artifacts and prompt guidance.

npx agentxchain init --governed --template web-app
cd my-agentxchain-project
agentxchain status

agentxchain status [--json]

Use this first when the workflow feels opaque. For governed projects it renders template, phase, run status, active turns, gate status, budget, and any blocked-state recovery guidance. With --json, it exposes the structured state for scripts or dashboards, including a top-level template field in addition to config.template.

agentxchain resume [--role <role>] [--turn <id>]

Writes or rewrites the dispatch bundle and exits. If the run is idle, it initializes a new run first. If the run is paused with a failed or retrying retained turn, it redispatches that same turn instead of inventing a new one.

# assign the planning role without waiting
agentxchain resume --role pm

# redispatch a retained turn after operator recovery
agentxchain resume --turn turn_abc123

agentxchain step [--role <role>] [--resume] [--turn <id>]

This is the truthful one-turn command. It can assign new work, continue waiting on an active turn, or redispatch a retained blocked turn when paired with --resume. It never loops forever and it never silently advances paused approvals.

  • manual runtimes print instructions and poll for the staged result file
  • local_cli runtimes spawn the configured subprocess and then validate the staged result
  • api_proxy runtimes perform the API call synchronously and stage the provider-backed result before validation

agentxchain accept-turn [--turn <id>] [--resolution standard|human_merge]

Accepts a staged result into governed history. Acceptance re-validates the JSON, records accepted decisions, updates orchestrator state, appends to TALK.md, and clears the targeted staging file. If multiple active turns exist, --turn becomes mandatory.

agentxchain reject-turn [--turn <id>] --reason "..." [--reassign]

Rejects the current staged result. Before retry exhaustion it keeps the same turn_id, increments the governed attempt count, and rewrites the dispatch bundle. For conflicted turns, --reassign is the fast recovery path that preserves the conflict context while redispatching the same turn.

Approvals and recovery

  • If the run is paused for approval, do not keep calling step. Use approve-transition or approve-completion.
  • If multiple active turns exist, targeted operations such as step --resume, accept-turn, and reject-turn require --turn <id>.
  • If a turn is conflicted, the recovery commands are explicit: reject-turn --turn <id> --reassign or accept-turn --turn <id> --resolution human_merge.

approve-transition

Only succeeds when the run is paused with a pending phase transition. On success it advances the phase, clears the blocker, and leaves the run ready for the next step or resume.

approve-completion

Only succeeds when QA or another review role requested run completion and the run is paused awaiting human approval. On success it marks the run completed and records the completion timestamp.

Blocked-state mental model

A blocked run is not the same as a paused approval. Blocked means the orchestrator retained a turn because recovery is required. Use status first, then follow the printed recovery command instead of guessing.

Validation and migration

validate --mode kickoff|turn|full

Use kickoff to check governed project wiring, turn to validate a staged turn result, and full to run both sets. In turn mode, a missing staged result warns instead of failing hard.

migrate [--yes] [--json]

Converts a legacy v3 project into the governed format. This is the bridge for existing users, not the recommended starting point for new projects.

Dashboard

agentxchain dashboard [--port <port>] [--no-open]

Starts a read-only local bridge server and opens the governance dashboard in your browser. The dashboard is an observation surface, not a control plane: gate approvals and turn acceptance remain CLI-only. It binds to 127.0.0.1 only — no network exposure.

Requires an initialized governed project (.agentxchain/ directory must exist). Default port is 3847. The bridge watches both repo-local state and .agentxchain/multirepo/ coordinator state when present, then pushes live invalidation events over WebSocket so the browser view stays current without manual refresh.

agentxchain dashboard
agentxchain dashboard --port 4000
agentxchain dashboard --no-open

Seven views

Initiative

The coordinator overview for multi-repo work. Shows super_run_id, coordinator status and phase, repo run linkage, barrier snapshot, recent barrier transitions, and any pending coordinator gate. The exact CLI action shown for coordinator gates is agentxchain multi approve-gate.

Cross-Repo Timeline

Renders coordinator history newest-first across repos: dispatches, acceptance projections, context generation, resync events, and gate requests/approvals. Unknown coordinator event types are still shown as generic entries instead of being dropped.

Run Timeline

The primary view. Renders the current governed run as a vertical timeline of accepted turns — newest first. Each turn card shows the agent role, summary, files changed, decisions, objections, risks, and verification evidence. Status badges are color-coded: running (green), paused (yellow), blocked (red).

Decision Ledger

Renders decision-ledger.jsonl as a filterable table with Turn, Agent, and Decision columns. v2.0 provides a role dropdown and a text query filter. Phase filters, date range, and objection presence filters are deferred to v2.1+.

Hook Audit Log

Renders hook-audit.jsonl as an event stream: phase, hook name, verdict (allow / warn / block), orchestrator action, and duration. Also shows hook annotations from hook-annotations.jsonl. No client-side filters in v2.0.

Blocked State

When the run is in blocked status, this panel shows the block reason, which agent or hook caused it, and the exact CLI recovery command to run next. It also surfaces nearby hook/validation audit context when available. Recovery commands have click-to-copy support.

The panel reads state.blocked_reason.recovery as the runtime source of truth. It does not show a history of prior blocks — no dedicated blocked-event ledger exists yet in v2.0.

When coordinator state exists and repo-local state does not, the same view falls back to coordinator blocked state plus recent coordinator hook audit context from .agentxchain/multirepo/hook-audit.jsonl.

Gate Review

When a human gate is pending (phase transition or run completion), this panel shows aggregated evidence from all turns since the last phase boundary — not just the requesting turn. It renders per-agent summaries, collected objections, risks, decisions, and files changed. The exact CLI command (agentxchain approve-transition or agentxchain approve-completion) is shown with click-to-copy.

When coordinator state has a pending gate, this view aggregates coordinator acceptance projections and barrier state, then renders agentxchain multi approve-gate.

The dashboard does not approve gates itself. The CLI remains the only mutation path.

What the dashboard does NOT do in v2.0

  • No write operations — cannot approve gates, accept turns, or resume runs
  • No dispatch bundle inspection — prompt/assignment content is not rendered
  • No git diff rendering — file paths only, no inline diffs
  • No expandable full turn JSON or per-turn hook detail
  • No desktop notifications — use the hook system (on_escalation) for alerting
  • No cloud-hosted state or multi-tenant persistence — this remains a localhost observation surface

Plugins

Plugins package hook integrations with a manifest, install lifecycle, and clean removal. See the full plugin docs for authoring details.

plugin install <source>

Installs a plugin from a local directory, .tgz archive, or npm package spec. Validates the agentxchain-plugin.json manifest, enforces config_schema when present, checks for hook-name collisions, copies the payload into .agentxchain/plugins/, rewrites relative command paths, and merges hooks into agentxchain.json.

agentxchain plugin install ./my-plugin
agentxchain plugin install ./plugins/plugin-slack-notify
agentxchain plugin install ./plugins/plugin-json-report

plugin list [--json]

Lists installed plugins with name, version, install path, source type, and hook bindings by phase.

plugin upgrade <name> [source]

Stages and validates the new version, swaps the install atomically, and rolls back to the previous plugin if config commit fails.

plugin remove <name>

Removes only the named plugin's hook bindings and metadata. Unrelated operator hooks are preserved. The install directory under .agentxchain/plugins/ is deleted.

Common sequences

Manual-first planning turn

npx agentxchain init --governed
cd my-agentxchain-project
agentxchain step --role pm

# fill .agentxchain/staging/<turn_id>/turn-result.json
agentxchain validate --mode turn
agentxchain accept-turn
agentxchain approve-transition

Recover a conflicted turn

agentxchain status
agentxchain reject-turn --turn turn_abc123 --reassign

# or after a human merge:
agentxchain accept-turn --turn turn_abc123 --resolution human_merge

Continue after a retained failed turn

agentxchain status
agentxchain resume --turn turn_abc123

# or run the full lifecycle again
agentxchain step --resume --turn turn_abc123
