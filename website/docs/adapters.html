Docs / Adapters

Three adapters, one contract. Governance stays the same regardless of how you reach the agent.

Adapters are the transport layer between the orchestrator and agent runtimes. Each adapter is intentionally thin (~200 lines). The orchestrator handles governance — dispatch bundles, validation, gates, state. Adapters handle delivery — getting the prompt to the agent and the result back to staging.

The shared adapter contract

Every adapter — regardless of transport — follows the same 3-phase lifecycle:

dispatch  →  wait  →  collect
   │            │          │
   ▼            ▼          ▼
Deliver      Block      Read staged
prompt to    until      turn-result.json
the agent    result     and return to
             appears    orchestrator

Adapters never write state. Only the orchestrator advances .agentxchain/state.json. Adapters never validate turn results (that's the orchestrator's 5-stage validator). Adapters never decide turn routing (agents propose proposed_next_role, orchestrator decides).

Filesystem contract

The orchestrator writes the dispatch bundle. The agent (via adapter) writes the staging result. Everything else is off-limits.

# Orchestrator writes (adapter reads)
.agentxchain/dispatch/turns/<turn_id>/ASSIGNMENT.json
.agentxchain/dispatch/turns/<turn_id>/PROMPT.md
.agentxchain/dispatch/turns/<turn_id>/CONTEXT.md

# Agent writes via adapter (orchestrator reads)
.agentxchain/staging/<turn_id>/turn-result.json

# Reserved — adapters MUST NOT write
.agentxchain/state.json
.agentxchain/history.jsonl
.agentxchain/decision-ledger.jsonl
.agentxchain/lock.json

What the dispatch bundle contains

The orchestrator renders three files per turn:

  • ASSIGNMENT.json — Machine-readable envelope: run ID, turn ID, phase, role, write authority, deadline, allowed next roles, budget reservation, and conflict context (for parallel turns).
  • PROMPT.md — Rendered seed prompt with the agent's mandate, protocol rules (challenge requirement, reserved paths, output format), write authority rules, phase exit gate requirements, and a full JSON template for the turn result.
  • CONTEXT.md — Execution context: current state summary, last accepted turn summary, blockers, and gate status.

What the staged result must look like

The turn result is a JSON file with required fields validated by a 5-stage pipeline (schema → assignment → artifact → verification → protocol compliance). The key fields:

{
  "schema_version": "1.0",
  "run_id": "...",
  "turn_id": "...",
  "role": "qa",
  "runtime_id": "qa_review",
  "status": "completed",
  "summary": "Reviewed auth middleware...",
  "decisions": [{ "id": "DEC-12", "text": "...", "category": "..." }],
  "objections": [{ "id": "OBJ-5", "text": "...", "severity": "..." }],
  "files_changed": [".planning/acceptance-matrix.md"],
  "verification": { "status": "pass", "evidence_summary": "..." },
  "artifact": { "type": "review" },
  "proposed_next_role": "human"
}

Review-only roles must raise at least one objection. This is the mandatory challenge requirement — blind agreement is a protocol violation.

manual — Human-operated turns

Use when: Learning the protocol for the first time, PM turns where a human authors the planning artifacts, or any role where you want full control over the output.

How it works

  1. Dispatch: step --role pm writes the dispatch bundle and prints a formatted instruction box showing the paths to the prompt, context, and staging target.
  2. Wait: The CLI polls .agentxchain/staging/<turn_id>/turn-result.json every 2 seconds, waiting for you to create the file. Default timeout: 20 minutes.
  3. Collect: Once the file exists and is >2 bytes, the adapter reads it and hands it to the orchestrator for validation.

Your workflow

# 1. Start the turn
agentxchain step --role pm

# 2. Read the dispatch bundle
cat .agentxchain/dispatch/turns/<turn_id>/PROMPT.md

# 3. Do the work — edit planning files, write code, whatever the role requires

# 4. Create the staged result (copy the JSON template from PROMPT.md)
$EDITOR .agentxchain/staging/<turn_id>/turn-result.json

# 5. Optionally validate before accepting
agentxchain validate --mode turn

# 6. Accept
agentxchain accept-turn

Configuration

// agentxchain.json → runtimes
{
  "pm_manual": {
    "type": "manual",
    "roles": ["pm"]
  }
}

No additional fields needed. The manual adapter has no subprocess, no API call, no auth — it's pure filesystem handoff.

local_cli — Subprocess-based automation

Use when: Automating dev turns with local coding agents (Claude Code, Codex CLI, Aider, or any CLI tool that reads a prompt and writes files).

How it works

  1. Dispatch: Spawns the configured command as a subprocess, delivering the rendered prompt via one of three transport modes.
  2. Wait: Blocks until the subprocess exits. Collects stdout/stderr for audit logs.
  3. Collect: Reads the staged turn-result.json. A process exit code of 0 does not mean the turn succeeded — only a valid staged result counts.

Prompt transport modes

argv

Replaces {prompt} placeholders in the command/args with the full rendered prompt. Default if {prompt} appears anywhere in the command.

"command": ["claude", "--print", "-p", "{prompt}"]

stdin

Writes the full prompt to the subprocess's stdin. Use when the tool reads from stdin or the prompt is too large for argv.

"command": ["my-agent"],
"prompt_transport": "stdin"

dispatch_bundle_only

No prompt delivery — the subprocess reads directly from .agentxchain/dispatch/turns/<turn_id>/PROMPT.md and CONTEXT.md. Use for agents that are already dispatch-bundle-aware.

"command": ["my-governed-agent", "--dispatch-dir", ".agentxchain/dispatch/turns"],
"prompt_transport": "dispatch_bundle_only"

Timeout handling

Configurable via timeout_seconds (default: 120). On timeout:

  1. Sends SIGTERM to the subprocess
  2. Waits 10 seconds for graceful shutdown
  3. Sends SIGKILL if still running

Logs are persisted to .agentxchain/dispatch/turns/<turn_id>/stdout.log for post-mortem debugging.

Configuration

// agentxchain.json → runtimes
{
  "dev_claude": {
    "type": "local_cli",
    "roles": ["dev"],
    "command": ["claude", "--print", "-p", "{prompt}"],
    "cwd": ".",
    "timeout_seconds": 300
  }
}

api_proxy — Direct model API calls

Use when: QA/review turns where the model reviews code and produces structured feedback. In v1, API proxy is restricted to review_only roles — it cannot make authoritative file changes.

How it works

  1. Dispatch: Sends an HTTP POST to the model provider (Anthropic Messages API in v1) with the rendered prompt and context as the user message.
  2. Wait: Synchronous — the adapter blocks on the HTTP response.
  3. Collect: Extracts the turn result JSON from the response text (tries full-text parse, then markdown fences, then JSON boundary detection). Stages the result and audit artifacts.

Retry logic

Exponential backoff with jitter. Retries on transient errors only — auth failures, invalid requests, and context overflow are not retried.

// Default retry policy
{
  "max_attempts": 3,
  "base_delay_ms": 1000,
  "max_delay_ms": 8000,
  "backoff_multiplier": 2,
  "jitter": "full"
}

Every retry attempt is recorded in .agentxchain/staging/retry-trace.json with timestamps, outcomes, delays, and per-attempt token usage.

Preflight tokenization

Optional. When enabled, the adapter estimates token usage before making the API call. If the prompt + context exceeds the model's context window (minus safety margin), the adapter returns context_overflow without spending money on a doomed request.

"preflight_tokenization": {
  "enabled": true,
  "tokenizer": "provider_local",
  "safety_margin_tokens": 2048,
  "context_window_tokens": 200000
}

Cost tracking

Provider telemetry is authoritative. The adapter reads input_tokens and output_tokens from the API response and computes cost using published rates. This overwrites any cost the model may have self-reported in the turn result.

Supported models (v1)

claude-sonnet-4-6      $3.00 / $15.00 per 1M tokens (in/out)
claude-opus-4-6       $15.00 / $75.00 per 1M tokens (in/out)
claude-haiku-4-5      $0.80 / $4.00 per 1M tokens (in/out)

Audit artifacts

The API proxy writes more audit artifacts than any other adapter — all best-effort (failure to write doesn't fail the turn):

  • API_REQUEST.json — Request metadata (provider, model, endpoint, params, timestamp). No API key.
  • TOKEN_BUDGET.json — Preflight token budget report (if tokenization enabled).
  • CONTEXT.effective.md — Exact context sent after any compression.
  • provider-response.json — Raw API response (success and extraction failure).
  • api-error.json — Classified error object (on failure).
  • retry-trace.json — Full retry history with per-attempt telemetry.

Error classification

Every API failure is classified into one of 13 error classes, each with a recovery hint and retryability flag:

Retryable:        rate_limited, provider_overloaded, timeout,
                  network_failure, response_parse_failure,
                  turn_result_extraction_failure, unknown_api_error

Not retryable:    auth_failure, model_not_found, context_overflow,
                  invalid_request, missing_credentials,
                  unsupported_provider

Configuration

// agentxchain.json → runtimes
{
  "qa_api": {
    "type": "api_proxy",
    "roles": ["qa"],
    "provider": "anthropic",
    "model": "claude-sonnet-4-6",
    "auth_env": "ANTHROPIC_API_KEY",
    "max_output_tokens": 4096,
    "timeout_seconds": 120,
    "retry_policy": {
      "enabled": true,
      "max_attempts": 3
    },
    "preflight_tokenization": {
      "enabled": true,
      "safety_margin_tokens": 2048,
      "context_window_tokens": 200000
    }
  }
}

Implementing a new adapter

Adding support for a new agent runtime is a ~200-line adapter. Here's what you need.

The contract

  1. Read the dispatch bundle. Load ASSIGNMENT.json, PROMPT.md, and CONTEXT.md from .agentxchain/dispatch/turns/<turn_id>/.
  2. Deliver the prompt to the agent runtime however it expects (HTTP, subprocess, message queue, WebSocket — the transport is yours to define).
  3. Stage the result. Write the agent's turn result to .agentxchain/staging/<turn_id>/turn-result.json. Must be valid JSON, >2 bytes.
  4. Return a status object: { ok: boolean, error?: string }. Optionally include classified error details with error_class, message, recovery, and retryable.

Rules

  • Never write to .agentxchain/state.json, history.jsonl, decision-ledger.jsonl, or lock.json.
  • Never validate the turn result — the orchestrator's 5-stage validator handles that.
  • Never decide turn routing — the agent proposes proposed_next_role, the orchestrator decides.
  • Respect write_authority from the assignment. If review_only, reject authoritative roles.
  • Support options.signal (AbortSignal) for cancellation.
  • Support timeout via deadline_at from the assignment or a configurable default.
  • Audit artifacts are best-effort. Don't fail the turn because you couldn't write a log file.

Minimal implementation shape

// my-adapter.js
export async function dispatchMyAdapter(root, state, config, options = {}) {
  const turn = state.active_turns?.[0] ?? state.current_turn;
  if (!turn) return { ok: false, error: 'No active turn in state' };

  // 1. Read dispatch bundle
  const prompt = readFileSync(getDispatchPromptPath(turn.turn_id), 'utf8');
  const context = readFileSync(getDispatchContextPath(turn.turn_id), 'utf8');

  // 2. Send to your runtime (HTTP, subprocess, etc.)
  const response = await yourRuntime.send({ prompt, context });

  // 3. Stage the result
  const stagingDir = getTurnStagingDir(turn.turn_id);
  mkdirSync(stagingDir, { recursive: true });
  writeFileSync(join(stagingDir, 'turn-result.json'),
    JSON.stringify(response.turnResult, null, 2));

  return { ok: true };
}

Troubleshooting

Subprocess exits 0 but the turn fails

A clean exit code does not mean the turn succeeded. The orchestrator validates the staged turn-result.json independently. Check that the file exists, is valid JSON, matches the required schema, and satisfies protocol rules (mandatory challenge for review roles, correct role/turn assignment).

API proxy returns context_overflow

The prompt + context exceeds the model's context window. Either reduce context (shorter history, fewer files), increase context_window_tokens if using a larger model, or switch to a model with a bigger window.

API proxy returns auth_failure

The environment variable specified in auth_env is missing or contains an invalid key. Check echo $ANTHROPIC_API_KEY (or whatever your auth_env points to).

Manual adapter times out

The default polling timeout is 20 minutes. If you need more time, cancel the current step and re-run it. The dispatch bundle persists — you don't lose work.
