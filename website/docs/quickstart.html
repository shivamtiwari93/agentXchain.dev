Run a governed PM → dev → QA flow without guessing the protocol.

This page documents the workflow that exists today. Governed mode is staged and gate-driven: step creates a turn, the role produces a structured result, the orchestrator validates it, and a human approves phase changes and final ship decisions.

Before you start

Use this quickstart if you want the governed workflow, not the older lock-based IDE flow. All commands below are governed commands.

  • Node.js 18.17+ or 20.5+
  • git available, because governed turns compare against repository state
  • Either npx agentxchain or a global install via npm install -g agentxchain
  • The default scaffold creates a new folder named my-agentxchain-project/; governed commands run inside that folder, not beside it
  • You can make scaffold intent explicit with --template <id>; built-ins are generic, api-service, cli-tool, and web-app
  • The default QA runtime is api_proxy with ANTHROPIC_API_KEY; if you want a provider-free walkthrough, switch QA to manual before the QA step
  • Expectation set correctly: the first run is usually manual-first, because you are learning the turn-result contract
# reproducible scaffold: skip the interactive name prompt
npx agentxchain init --governed -y
cd my-agentxchain-project
git init
git add -A
git commit -m "initial governed scaffold"

# or install globally first
npm install -g agentxchain
agentxchain --version

If you want to choose the project name interactively, omit -y and answer the prompt. The rest of the workflow stays the same.

What governed mode actually creates

The scaffold is not cosmetic. These files are the protocol surface the orchestrator reads and validates.

Template choice is scaffold intent, not runtime magic. init --governed --template api-service adds project-type-specific planning files and prompt guidance. The built-ins are generic, api-service, cli-tool, and web-app.

Five-minute walkthrough

1 Initialize the governed project

npx agentxchain init --governed -y
cd my-agentxchain-project
git init
git add -A
git commit -m "initial governed scaffold"
agentxchain status --json

The initial state is idle in the planning phase. No run exists until you assign the first turn.

If you already know the repo shape, initialize with a template such as npx agentxchain init --governed --template web-app -y. status will then surface Template: web-app so operators can see the scaffold intent directly.

2 Assign the PM turn

agentxchain step --role pm

For a manual role, step writes a dispatch bundle, prints the exact PROMPT.md and turn-result.json paths, and then waits for the staged result. It does not invent the PM output for you.

Open the generated prompt bundle under .agentxchain/dispatch/turns/<turn_id>/. The exact turn id comes from the active run.

3 Create the PM artifacts and staged result

# edit the planning artifacts
$EDITOR .planning/ROADMAP.md
$EDITOR .planning/PM_SIGNOFF.md

# copy the JSON template from PROMPT.md into:
.agentxchain/staging/<turn_id>/turn-result.json

The staged turn result is the contract. It records summary, objections, files changed, decisions, and any requested phase transition. Do not freestyle this file. Copy the exact template from the dispatch bundle so run_id, turn_id, role, and runtime_id stay aligned.

{
  "schema_version": "1.0",
  "run_id": "run_...",
  "turn_id": "turn_...",
  "role": "pm",
  "runtime_id": "manual-pm",
  "status": "completed",
  "summary": "Defined the planning scope and acceptance criteria.",
  "decisions": [{ "id": "DEC-001", "category": "scope", "statement": "..." , "rationale": "..." }],
  "objections": [{ "id": "OBJ-001", "severity": "medium", "against_turn_id": "TODO", "statement": "...", "status": "raised" }],
  "files_changed": [],
  "verification": { "status": "skipped", "commands": [], "evidence_summary": "Review turn.", "machine_evidence": [] },
  "artifact": { "type": "review", "ref": null },
  "proposed_next_role": "human",
  "phase_transition_request": "implementation",
  "run_completion_request": null
}
# optional preflight before acceptance
agentxchain validate --mode turn

4 Accept the PM turn and pass the planning gate

agentxchain accept-turn
agentxchain approve-transition

If the PM turn requested planning -> implementation and the required files exist, the orchestrator pauses for human approval. That pause is intentional. Humans control phase transitions.

5 Run implementation

agentxchain step --role dev

The default example uses a local_cli dev runtime. AgentXchain dispatches the turn, waits for the role output, validates the staged result, and expects real verification evidence before the implementation gate can pass.

6 Run QA and complete the run

# default path: keep api_proxy QA and set ANTHROPIC_API_KEY
agentxchain step --role qa

# provider-free path: change api-qa.type to "manual", then:
agentxchain step --role qa

agentxchain approve-completion

QA is review-only. A ship-ready QA turn updates .planning/acceptance-matrix.md and .planning/ship-verdict.md, requests run completion, and then waits for human approval. Agents cannot self-ship. If you keep the default scaffold unchanged, this step requires ANTHROPIC_API_KEY.

Know which adapter you are using

manual

The operator reads the dispatch bundle, does the work, and stages turn-result.json manually. Best for learning the protocol and for human-authored turns.

local_cli

AgentXchain spawns a local coding agent such as Claude Code, Codex, or Aider through a configured command and then validates the result.

api_proxy

AgentXchain calls a model provider directly, stages the returned result, and applies the same validation and gate rules as any other turn. In v1, it is restricted to review_only roles such as QA.

What people usually get wrong

  • init --governed is interactive by default. Use -y when you want a reproducible copy-paste path.
  • step is not “ship the whole project.” It is one governed turn lifecycle.
  • The dispatch bundle and staged result paths are first-class protocol artifacts, not implementation clutter.
  • The default scaffold is not provider-free end to end. QA uses api_proxy unless you change it.
  • If a command stops because approval is pending, that is the governance model working correctly, not a broken run.

Troubleshooting

The run is waiting and I do not know why

Run agentxchain status. It shows the phase, active turns, pending approvals, and blocked recovery guidance. For a visual overview, run agentxchain dashboard to open the read-only governance dashboard in your browser.

accept-turn says validation failed

Inspect the staged file and rerun agentxchain validate --mode turn. The validator checks schema, role assignment, artifact claims, verification evidence, and gate compliance.

The staged result file does not exist

Re-open the latest dispatch bundle under .agentxchain/dispatch/turns/<turn_id>/. The matching staging target is .agentxchain/staging/<turn_id>/turn-result.json.

QA fails before it starts

The default scaffold sets QA to api_proxy with auth_env: ANTHROPIC_API_KEY. Export that key or switch the api-qa runtime in agentxchain.json to manual for a provider-free walkthrough.
