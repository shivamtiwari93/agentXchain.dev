Templates are scaffold intent, not protocol magic.

Use governed templates when you want project-shape-specific planning artifacts and prompt guidance without changing the protocol contract. The template lives in agentxchain.json, not hidden state.

What a template actually is

Template choice is scaffold intent. It adds planning structure and guidance, then records that choice in the top-level template field in agentxchain.json. It does not fork the protocol, gates, or turn-result schema.

  • Templates are for governed mode only.
  • Existing governed repos without a template field are treated as generic.
  • The scaffold decision is config, not run state.
  • Template choice stays visible in status and status --json.

Built-in templates

Template Best fit Planning artifacts added
generic Default governed scaffold No extra project-type files
api-service Backend services and API-heavy repos api-contract.md, operational-readiness.md, error-budget.md
cli-tool Command-line tools and installable CLIs command-surface.md, platform-support.md, distribution-checklist.md
web-app Browser-facing products and UI-heavy repos user-flows.md, ui-acceptance.md, browser-support.md

If none of the specialized templates fit honestly, stay on generic. Decorative metadata is useless. Clear planning artifacts are the point.

Choose a template at init time

The init-time command surface is agentxchain init --governed --template <id>.

npx agentxchain init --governed --template web-app -y
cd my-agentxchain-project

This writes the governed scaffold plus the template-specific planning files. The config then records a top-level template field so later status, migrations, and docs can see the scaffold intent directly.

Inspect the shipped template surface

agentxchain template list
agentxchain template list --json

template list is the discovery command. It prints the built-in descriptions, planning artifact filenames, prompt-override roles, and acceptance hints. Use it before mutating an existing repo if you want the CLI to show the current shipped surface instead of guessing.

Apply template guidance to an existing repo

The mutation command surface is agentxchain template set <id>.

agentxchain template set api-service --yes
agentxchain template set web-app --dry-run

template set is intentionally conservative. It is additive, not destructive.

What it does

  • Updates the config template field
  • Creates missing planning artifacts from the target template
  • Appends ## Project-Type-Specific Guidance once per prompt when missing
  • Appends ## Template Guidance once to .planning/acceptance-matrix.md when missing
  • Appends a template_set entry to .agentxchain/decision-ledger.jsonl

What it does not do

  • Overwrite existing planning docs
  • Replace an existing prompt-guidance section
  • Delete old template files when switching templates
  • Infer a template from repo contents

That means switching from one non-generic template to another is still additive. If you want to replace prior guidance, do it manually and deliberately.

Template intent stays visible

agentxchain status
agentxchain status --json

Human-readable status prints Template: web-app. JSON status includes both a top-level template field and config.template. That keeps scaffold intent visible to operators and automation without burying it in run state.

Governed multi-agent software delivery. Planning structure is explicit on purpose because long-horizon agent work collapses without artifact discipline.
