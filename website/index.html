Philosophy

Coordination, not orchestration

Most AI agent systems have one brain that spawns sub-agents. We think that’s wrong. Real software isn’t built by one mind delegating — it’s built by a team that argues, challenges, and converges.

No central decision-maker

No orchestrator agent deciding what everyone does. Each agent has its own mandate, its own perspective, its own judgment. They coordinate through shared state, not a boss.

Divergent agents, healthy friction

The PM pushes for user value. The dev pushes for feasibility. QA pushes for correctness. The eng director pushes for quality. They’re supposed to disagree. That tension produces better software than any single agent ever could.

Real-world pressure to ship

Phases, turn limits, mandatory verification, human checkpoints. The team can’t bikeshed forever. Constraints force convergence on a real product — just like a deadline does for human teams.

How real teams work, applied to AI

In a real software company, the PM doesn’t write code. The dev doesn’t decide what to build. QA doesn’t trust anyone else’s test results. And there’s always a moment where someone says “this isn’t good enough” and sends it back.

That’s what AgentXchain recreates. Not the org chart. Not the meetings. The distributed decision-making — where no single agent has the full picture, but the team collectively does.

Prompts and protocol text require agents to challenge prior work. The CLI enforces locks, handoffs, and artifact checks; model behavior still depends on following those rules.

How it works

PM-first by default. Execution only after signoff.

The workflow is optimized for product quality, not raw speed.

1

Align

Run agentxchain kickoff. PM interviews you, clarifies scope, and writes explicit acceptance criteria.

2

Gate

Kickoff must pass validation: PM signoff, waves/phases, planning docs, and QA structure all in place.

3

Execute

Remaining agents run with explicit turns, human checkpoints, and evidence-driven QA.

4

Decide

Ship when quality gates pass. Block and route to human when validation fails.

Quality

QA that actually catches things

Structured QA artifacts and agent mandates (not an automated test runner). Teams configure checks; agents are prompted to produce evidence, acceptance matrices, and explicit ship readiness in docs.

Structured bug reports

Every bug: ID, severity (P0–P3), exact repro steps, expected vs actual, file and line. No vague “something seems off.”

Comprehensive UX audit

First impressions, core flow, forms, responsive at 375/768/1440px, WCAG accessibility, error states, visual consistency.

Ship readiness in the record

QA role is prompted to state YES / YES WITH CONDITIONS / NO + blockers in team docs each turn. Outcomes depend on agents following the protocol.

Project structure

Agents maintain their own documentation

Every project gets a .planning/ folder with structured docs. PM writes requirements. QA maintains test coverage. Eng Director writes reviews. All machine-readable, all git-tracked.

PROJECT.md + REQUIREMENTS.md

Vision, constraints, numbered requirements with acceptance criteria. Created by the PM on turn 1.

ROADMAP.md + phases/

Phased delivery. Each phase gets plans, reviews, tests, bugs. Agents fill these as work progresses.

qa/ folder

Test coverage map, acceptance matrix, UX audit checklist, bug log, regression log. QA maintains all of it.

Product outcomes

PM-first signoff prevents teams from coding before the problem is clear.
Higher quality releases
