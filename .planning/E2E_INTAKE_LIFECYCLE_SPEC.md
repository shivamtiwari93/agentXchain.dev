# E2E Intake Lifecycle Specification

> CLI-subprocess proof for the shipped intake lifecycle after V3-S5.

---

## Purpose

The intake lifecycle is now implemented across S1-S5, but the proof surface is still fragmented across command-level and library-level tests. That is not enough for a workflow-heavy feature. The missing proof is a single end-to-end test that drives the shipped intake surface as an operator would and closes the loop through the real governed acceptance path.

This slice proves one honest happy path:

`record -> triage -> approve -> plan -> start -> accept-turn -> resolve`

It must not fake completion by mutating `.agentxchain/state.json` directly after `intake start`. If the governed run outcome matters, the test must reach that outcome through the actual governed accept path.

---

## Interface

### Test File

- `cli/test/e2e-intake-lifecycle.test.js`

### Supporting Contract

- The test uses real CLI subprocesses for:
  - `agentxchain intake record`
  - `agentxchain intake triage`
  - `agentxchain intake approve`
  - `agentxchain intake plan`
  - `agentxchain intake start`
  - `agentxchain accept-turn`
  - `agentxchain intake resolve`
- The test may write a synthetic staged turn result under the turn-specific staging path because no live runtime is involved.
- The test may seed a minimal governed repo fixture directly on disk before the CLI sequence begins.

---

## Behavior

### Scenario

1. Create a temporary governed repo with git initialized and a clean baseline commit.
2. Configure a single final phase so the first accepted governed turn can complete the run without extra approval choreography.
3. Record a manual intake event through `agentxchain intake record --json`.
4. Triage the detected intent through `agentxchain intake triage --json`.
5. Approve the triaged intent through `agentxchain intake approve --json`.
6. Plan the approved intent through `agentxchain intake plan --json` using a governed template that emits real `.planning/` artifacts.
7. Commit the planning artifacts so `intake start` captures a clean product baseline for the governed turn.
8. Start the planned intent through `agentxchain intake start --json`.
9. Create a real product-file change in the worktree after the turn is assigned.
10. Write a valid staged turn result for the assigned turn with:
    - `status: "completed"`
    - `run_completion_request: true`
    - matching `run_id`, `turn_id`, `role`, and `runtime_id`
    - a declared `files_changed` entry matching the real worktree change
    - passing verification metadata
11. Accept the staged result through `agentxchain accept-turn`.
12. Resolve the linked intent through `agentxchain intake resolve --json`.

### Assertions

The test must prove all of the following:

1. Intake artifacts exist on disk under:
   - `.agentxchain/intake/events/`
   - `.agentxchain/intake/intents/`
2. Planning artifacts generated by `intake plan` exist on disk before `intake start`.
3. `intake start` returns a real `run_id` and `turn_id`, and the intent records `target_run`, `target_turn`, and `started_at`.
4. `accept-turn` completes the governed run through the real acceptance path; the governed state reaches `completed`.
5. `intake resolve` transitions the intent from `executing` to `completed`.
6. `intake resolve` creates `.agentxchain/intake/observations/<intent_id>/`.
7. The completed intent history contains the final lifecycle linkage entry referencing the governed `run_id`.

### Explicit Non-Scope

- no blocked-path coverage in this slice
- no failed-path coverage in this slice
- no `intake scan` coverage in this slice
- no live adapter/runtime invocation
- no release-gate or observation-evidence behavior beyond the shipped scaffold

---

## Error Cases

1. If the test reaches `completed` by editing `.agentxchain/state.json` directly after `intake start`, the proof is invalid. That is state forgery, not E2E.
2. If the staged turn result does not match the assigned turn identity, the test must fail rather than patch around the mismatch.
3. If the product-file change is not reflected in `files_changed`, the acceptance path should fail and the test should surface that mismatch.
4. If planning artifacts are not generated or not present at start time, the test is invalid because it is not proving the shipped `plan -> start` contract.

## Acceptance Tests

### AT-E2E-INTAKE-001: Full intake lifecycle happy path

Given a clean temporary governed repo, when the operator drives:

`intake record -> triage -> approve -> plan -> start -> accept-turn -> resolve`

then:

- the event and intent artifacts exist
- planning artifacts are generated
- the intent links to a real governed run and turn
- the governed run reaches `completed` through `accept-turn`
- the linked intent reaches `completed` through `intake resolve`
- the observation directory scaffold exists
- the completed intent history records the governed run linkage
