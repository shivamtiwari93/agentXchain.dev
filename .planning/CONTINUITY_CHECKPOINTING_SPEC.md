# Continuity / Checkpointing Spec

**Status:** Implemented (Turn 35)
**Scope:** single-repo governed runs first; coordinator parity is follow-on work

## Purpose

AgentXchain already has partial continuity proof:

- `resume` and `step --resume` preserve a run across fresh processes
- `restart` reconstructs a run from durable state
- `.agentxchain/session.json` is written at some governance boundaries
- multi-session E2E proves basic cross-process continuity

That is not yet a complete checkpointing contract.

The missing product truth is: **what exactly is checkpointed, when checkpoint truth advances, how recovery chooses between checkpoint metadata and governed state, and how restart behaves when the repo has drifted since the checkpoint was written.**

This spec defines the next continuity slice so long-horizon recovery is explicit instead of spread across `restart`, `resume`, session status, and ad hoc tests.

## Interface

### Durable artifacts

- `.agentxchain/state.json`
  - canonical governed run state
- `.agentxchain/history.jsonl`
  - append-only turn history
- `.agentxchain/decision-ledger.jsonl`
  - append-only decision history
- `.agentxchain/session.json`
  - latest checkpoint metadata for single-repo restart
- `.agentxchain/SESSION_RECOVERY.md`
  - operator-facing recovery briefing generated by `agentxchain restart`

### Commands

- `agentxchain restart [--role <role>]`
  - reconstruct from checkpoint + governed state when session context is gone
- `agentxchain resume`
  - continue a governed run when session context is still coherent
- `agentxchain step --resume`
  - recover retained/blocked turn execution

### Checkpoint schema

`.agentxchain/session.json` must capture enough metadata to explain and safely resume the last known governed position:

```json
{
  "session_id": "session_abcd1234",
  "run_id": "run_123",
  "started_at": "2026-04-10T00:00:00.000Z",
  "last_checkpoint_at": "2026-04-10T00:15:00.000Z",
  "checkpoint_reason": "turn_assigned",
  "run_status": "active",
  "phase": "implementation",
  "last_turn_id": "turn_004",
  "last_completed_turn_id": "turn_003",
  "active_turn_ids": ["turn_004"],
  "last_role": "dev",
  "pending_gate": null,
  "pending_run_completion": null,
  "blocked": false,
  "baseline_ref": {
    "git_head": "abc123...",
    "git_branch": "main",
    "workspace_dirty": false
  },
  "agent_context": {
    "adapter": "api_proxy",
    "dispatch_dir": ".agentxchain/dispatch/turns/turn_004"
  }
}
```

Not every field must be populated on day one, but the contract is that checkpoint metadata explains:

1. which run is being resumed
2. which phase and turn were current
3. whether human approval or blocked recovery was pending
4. what repo baseline the checkpoint observed

## Behavior

### 1. Ground truth precedence

- `state.json` is the source of truth for governed status, active turns, pending gates, blocked state, and run identity.
- `session.json` is recovery metadata, not authority.
- If `session.json` disagrees with `state.json`, the runtime must treat the checkpoint as stale metadata and continue from governed state, never the reverse.

### 2. When checkpoints are written

Checkpoint writes must happen at every state transition that materially changes recovery semantics:

1. turn assignment or run reactivation
2. accepted turn
3. persisted blocked state
4. approved phase transition
5. approved run completion
6. explicit restart/reconnect

The current write points on acceptance, phase approval, and completion are not enough. A run that dies immediately after assignment still needs truthful restart context.

Checkpoint writes remain non-fatal. Failing to write `session.json` must not corrupt governed execution, but the warning must be visible in debug logs and surfaced in recovery artifacts when possible.

### 3. Restart semantics

`agentxchain restart` is the checkpoint recovery command, not a synonym for `resume`.

Rules:

1. If the run is terminal, fail.
2. If the run is blocked, fail and surface the recovery action from governed state.
3. If the run has an active retained turn, reconnect to that turn instead of assigning a replacement.
4. If the run is paused or idle with no active turn, reactivate and assign the next turn from routing, unless `--role` overrides it.
5. If a phase gate or run completion approval is pending, `restart` must not bypass approval; it should write `SESSION_RECOVERY.md` and surface the correct next operator command.
6. `restart` must regenerate the dispatch bundle from durable governed state when a retained turn exists but session-local context is gone.

### 4. Repo-state conflict semantics

Checkpointing is not honest if it ignores repo drift.

When `restart` runs, it must compare the current repo state against checkpoint baseline metadata:

- `git HEAD`
- current branch
- dirty/clean workspace status

Behavior:

1. If `run_id` mismatches, warn and continue from `state.json`.
2. If the checkpoint says the workspace was clean but the repo is now dirty, record a continuity warning in `SESSION_RECOVERY.md` and status output.
3. If an active retained turn exists and the repo has drifted since the checkpoint, reconnect to the retained turn but do not silently assign a new turn on top of it.
4. If restart cannot reconstruct a truthful dispatch context because the retained-turn files or baseline are irrecoverably inconsistent, fail closed and tell the operator to inspect or recover manually before resuming.

The contract is fail-closed on ambiguous retained work, not optimistic reassignment.

### 5. Recovery artifact contract

`SESSION_RECOVERY.md` must explain:

- run identity
- checkpoint time and reason
- last known phase and turn
- pending approval/block state
- recent decisions/history summary
- any checkpoint-vs-repo drift warnings
- exact next action (`restart`, `resume`, `step --resume`, `approve-transition`, `approve-completion`)

## Error Cases

| Condition | Behavior |
|---|---|
| `session.json` missing | `restart` may continue from `state.json` if governed state is sufficient; warn that checkpoint metadata is missing |
| `session.json` invalid JSON | Warn, ignore checkpoint metadata, continue from `state.json` if safe |
| `session.json` and `state.json` `run_id` mismatch | Warn, treat checkpoint as stale, continue from governed state |
| `state.json` missing | Fail with no governed run found |
| terminal run | Fail without mutating checkpoint |
| blocked run | Fail and surface recovery action from governed state |
| retained active turn + drifted repo baseline | Reconnect with warning, or fail closed if dispatch context cannot be regenerated truthfully |
| checkpoint write failure | Non-fatal warning only; governed transition still completes |

## Acceptance Tests

- `AT-CC-001`: `resume` or `restart` writes a checkpoint when assigning a turn.
- `AT-CC-002`: `accept-turn` updates `session.json` with the new `last_completed_turn_id` and checkpoint timestamp.
- `AT-CC-003`: blocked-state persistence updates `session.json` so recovery artifacts know the run was blocked.
- `AT-CC-004`: `approve-transition` updates checkpoint phase metadata.
- `AT-CC-005`: `approve-completion` updates checkpoint state to terminal completion.
- `AT-CC-006`: `restart` reconnects to an abandoned active turn instead of replacing it.
- `AT-CC-007`: `restart` on paused/idle state assigns the next turn from governed routing.
- `AT-CC-008`: pending gate or run-completion approval is surfaced as approval work, not bypassed by restart.
- `AT-CC-009`: stale `session.json` `run_id` mismatch warns and defers to `state.json`.
- `AT-CC-010`: dirty-workspace drift since checkpoint is surfaced in `SESSION_RECOVERY.md` and status continuity output.
- `AT-CC-011`: irreconcilable retained-turn drift fails closed instead of silently assigning a new turn.

## Follow-On Decisions

- **Turn 36:** Coordinator continuity should not overload the single-repo `.agentxchain/session.json` surface. If coordinator checkpoint parity is added, it should use a dedicated `.agentxchain/multirepo/session.json` contract aligned to coordinator-specific recovery semantics rather than pretending repo-local and coordinator checkpoints are interchangeable.
- **Turn 36:** `run` should not emit checkpoints on every loop boundary by default. Checkpoints advance when governance state changes recovery semantics: assignment, acceptance, blocked persistence, gate approval, completion, or explicit restart/reconnect. Loop-boundary heartbeats would add churn without improving truthful recovery.
