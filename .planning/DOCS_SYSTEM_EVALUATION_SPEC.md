# Docs System Evaluation — AgentXchain.dev

## Purpose

Revalidate whether AgentXchain.dev should keep the current Docusaurus-based docs stack or migrate to another OSS docs system, per the OSS-first principle in `WAYS-OF-WORKING.md` section 6.

## Interface

### Inputs

- Repo-local docs implementation:
  - `website-v2/package.json`
  - `website-v2/docusaurus.config.ts`
  - `website-v2/sidebars.ts`
  - `.github/workflows/deploy-gcs.yml`
- Product requirements derived from current repo behavior:
  - filesystem and git-native docs authoring
  - static deploy to GCS
  - local/self-hostable search
  - docs versioning suitable for release-note history and evolving protocol docs
  - low migration risk while BUG-44/45/46 remain open pending tester verification
- Official-source candidate review:
  - Docusaurus official versioning docs
  - Nextra official docs/site
  - Starlight official docs/site
  - Docmost official site/GitHub

### Output

- Decision record `DEC-DOCS-SYSTEM-001`
- Candidate verdicts
- Re-open trigger if product requirements change materially

## Behavior

1. Audit the current local docs surface instead of reasoning from memory.
2. Treat these as hard requirements unless the product vision changes:
   - static-site deployability
   - repo-native Markdown/MDX workflow
   - self-hostable search
   - first-class versioned docs / release-history support
3. Compare credible OSS alternatives against those requirements using official sources only.
4. Keep Docusaurus unless a candidate materially improves the product on hard requirements and repays migration cost.

## Current Stack

- **Framework**: Docusaurus 3.10.0 with `future.v4: true`
- **Runtime**: React 19 + TypeScript config
- **Search**: `@easyops-cn/docusaurus-search-local` enabled with hashed docs-only index
- **Sidebar**: Release notes use autogenerated sidebar entries from `website-v2/docs/releases`
- **Sitemap**: Docusaurus sitemap enabled; no committed manual `static/sitemap.xml`
- **Deploy**: static build deployed through `.github/workflows/deploy-gcs.yml`
- **Custom surface area**: moderate theme/custom CSS, but no custom Docusaurus plugin that would create migration lock-in

## Candidate Evaluation

### Hard Requirement Summary

- AgentXchain needs versioned release/history docs today. This is not optional while the public docs surface continues to publish versioned release notes and evolving protocol/operator guidance.
- Better default visuals are not enough to justify migration.

### Candidate Matrix

| Candidate | Verdict | Why |
| --- | --- | --- |
| **Docusaurus 3.x** | **KEEP** | Official versioning support exists today. Current repo already uses the required features: static deploy, version-capable docs engine, local search, autogenerated release-note sidebar, sitemap. |
| **Docmost** | **DISCARD** | Wrong category. Official site/GitHub position it as a collaborative wiki/knowledge-management product, not a filesystem-native developer-docs generator. |
| **Nextra** | **DISCARD FOR NOW** | Official docs show strong i18n and Pagefind search, but no first-class docs-versioning workflow comparable to Docusaurus. `Multi-Docs` is not release-version management. |
| **Starlight** | **DISCARD FOR NOW** | Official docs show strong sidebar generation, search, and i18n, but no built-in versioned-docs system comparable to Docusaurus. Strong contender only if versioning stops being a hard requirement. |

### Candidate Notes

#### Docusaurus

- Official docs provide a real docs-versioning workflow.
- The repo already captures the main earlier pain points without migration:
  - release-note sidebar is autogenerated
  - sitemap generation is enabled
  - local search is enabled

#### Docmost

- Useful OSS product, but for team wiki / knowledge-base use cases.
- Migrating AgentXchain.dev docs to Docmost would replace a repo-native static-docs workflow with a collaborative wiki category mismatch.

#### Nextra

- Strengths from official docs/site: Pagefind-backed search, i18n, Next.js integration.
- Blocking weakness for AgentXchain today: no official first-class docs-versioning contract matching Docusaurus' release/version workflow.
- Inference: Nextra is viable only if AgentXchain deliberately abandons docs versioning as a product requirement.

#### Starlight

- Strengths from official docs/site: autogenerated sidebars, site search, i18n, good static-site ergonomics.
- Blocking weakness for AgentXchain today: no official first-class docs-versioning system matching Docusaurus' current release/version contract.
- Inference: Starlight becomes worth revisiting only if release-history/versioned-doc requirements materially shrink.

## Decision

**DEC-DOCS-SYSTEM-001**: AgentXchain stays on Docusaurus 3.x.

### Rationale

1. **The current stack already satisfies the hard requirements.**
2. **The migration case is weak.** None of the OSS alternatives clearly beats Docusaurus on the current hard requirements.
3. **Versioning is the decisive separator.** Docusaurus has an official answer; the alternatives reviewed here do not.
4. **The earlier pain points were operational, not architectural.** Search, sitemap, and release-note navigation are already handled inside the current stack.
5. **OSS-first is already satisfied.** Docusaurus is the mature OSS baseline; switching OSS tools without net gain is churn, not strategy.

## Error Cases

- Do not reopen this evaluation because another tool has better default aesthetics.
- Do not treat `Multi-Docs`, i18n, or search alone as a substitute for first-class versioned docs.
- Do not start a docs-stack migration while BUG-44/45/46 remain open pending tester verification unless Docusaurus itself becomes a blocker.
- Reopen the evaluation only if one of these becomes true:
  - versioned docs stop being a hard requirement
  - Docusaurus stops meeting the static/self-hosted deployment model
  - a candidate ships official versioned-doc support that materially changes the tradeoff
