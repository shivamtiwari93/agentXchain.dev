# Docs System Evaluation — AgentXchain.dev

## Purpose

Evaluate whether the current Docusaurus-based docs stack should be replaced with an OSS alternative, per the OSS-first principle in WAYS-OF-WORKING.md §6.

## Current Stack

- **Framework**: Docusaurus 3.10.0 (MIT, React 19, TypeScript config)
- **Pages**: ~150 MDX files (core docs, 83 release notes, 14 examples, 24 integrations)
- **Custom components**: Minimal (1 reusable section renderer, 1 homepage)
- **Theme**: Custom CSS (~908 lines), Inter/JetBrains Mono fonts, dark-mode default
- **Plugins**: None custom — preset-classic only
- **Deploy**: GCS static site via `.github/workflows/deploy-gcs.yml`
- **Search**: None configured (no Algolia, no local search plugin)
- **Sitemap**: Manual static file (disabled auto-generation) — **pain point**
- **Sidebar**: Fully manual `sidebars.ts` with 83 hardcoded release entries — **pain point**

## Candidates Evaluated

### 1. Docmost (AGPL-3.0)

**Verdict: DISCARD — wrong category.**

Docmost is a wiki/knowledge-base (Notion/Confluence alternative), not a developer docs generator. No MDX, no filesystem-driven model, no versioning, no static deployment, no code syntax highlighting. The migration would be a conceptual mismatch, not just a technical one.

### 2. Mintlify (Proprietary SaaS)

**Verdict: DISCARD — not OSS, not self-hostable.**

Excellent API docs and beautiful defaults, but closed-source, vendor-locked, and costs money at scale. For a project that values governance and sovereignty, depending on a SaaS for the docs surface is a philosophical mismatch.

### 3. GitBook (Proprietary SaaS)

**Verdict: DISCARD — not OSS, not self-hostable.**

The old GitBook v1 OSS CLI is abandoned since 2019. Current GitBook is fully proprietary. No MDX, no git-native workflow.

### 4. Nextra (MIT, Next.js)

**Verdict: DISCARD — no built-in versioning.**

Clean DX with filesystem-based auto-sidebar via `_meta.json`. Built-in flexsearch. But no versioning support — dealbreaker for a protocol with 73+ versioned releases. Migration effort would be medium, but we'd lose a capability we already have.

### 5. Starlight / Astro (MIT)

**Verdict: DISCARD — no built-in versioning.**

Best built-in search (Pagefind), fastest build times (partial hydration), beautiful defaults. But no production-grade versioning. If we ever drop versioning needs, Starlight would be the strongest contender. Today it loses on a hard requirement.

## Decision

**DEC-DOCS-SYSTEM-001**: AgentXchain stays on Docusaurus 3.x.

### Rationale

1. **No candidate solves a problem Docusaurus doesn't.** The two real pain points (manual sidebar, manual sitemap) are Docusaurus configuration issues, not framework limitations.
2. **Versioning is non-negotiable.** Only Docusaurus has built-in, production-grade docs versioning. Every other viable OSS option (Nextra, Starlight) lacks it.
3. **Migration cost is real.** ~150 MDX files, custom CSS, GCS deploy pipeline, React components. Medium-to-high effort for zero capability gain.
4. **Search gap is closable.** Adding `docusaurus-search-local` or Algolia DocSearch (free for OSS) is a 15-minute fix, not a migration justification.
5. **OSS-first principle is satisfied.** Docusaurus *is* the mature OSS option. The principle says "prefer mature OSS over custom" — it doesn't say "keep switching between OSS options."

### What to fix instead

Rather than migrating, fix the real pain points in the current stack:

1. **Automate release notes sidebar**: ✅ Done — `sidebars.ts` uses `type: 'autogenerated'` with `dirName: 'releases'`.
2. **Re-enable auto-sitemap**: ✅ Done — `sitemap` config is active in `docusaurus.config.ts`, no manual `static/sitemap.xml`.
3. **Add offline search**: ✅ Done (Turn 163) — `@easyops-cn/docusaurus-search-local` added with hashed index, docs-only scope, and search term highlighting. Zero external service dependency.
