# Verification Pass Semantics Prompt Spec

## Purpose

Prevent authoritative implementation turns from staging validator-invalid passing results when they exercise expected failure paths as standalone non-zero shell commands.

## Interface

- Generic developer prompt generated by `cli/src/commands/init.js`
- Runtime dispatch prompt generated by `cli/src/lib/dispatch-bundle.js`
- Shipped governed todo example prompt and docs:
  - `examples/governed-todo-app/.agentxchain/prompts/dev.md`
  - `examples/governed-todo-app/README.md`
  - `website-v2/docs/quickstart.mdx`
- Verification-stage validator error text in `cli/src/lib/turn-result-validator.js`

## Behavior

The product must state the same rule everywhere an operator or agent is likely to rely on it:

1. `verification.status: "pass"` is only valid when every listed `verification.machine_evidence[].exit_code` is `0`.
2. Expected-failure checks must not appear as raw non-zero commands on a passing turn.
3. If an agent wants to prove an error path while still reporting `pass`, it must wrap that check in a test harness or shell assertion whose own exit code is `0` only when the failure occurs as expected.
4. Example/operator docs should prefer a single reproducible verification command such as `npm test` or `node --test` over loose ad hoc command lists.

## Error Cases

| Case | Required behavior |
|------|-------------------|
| Agent reports `verification.status: "pass"` with non-zero `machine_evidence` exit codes | Validator rejects the staged result and the error text explains the expected-failure wrapper rule |
| Prompt/docs mention only “report exit codes honestly” without naming the expected-failure trap | This is considered incomplete guidance and must fail content tests |
| Example docs imply that raw negative-case commands are acceptable proof on a passing turn | This is product drift and must fail content tests |

## Acceptance Tests

- **AT-VPS-001**: Scaffolded developer prompt states that expected-failure checks must be wrapped in a zero-exit test harness if the turn reports `verification.status: "pass"`.
- **AT-VPS-002**: Runtime dispatch prompt for an authoritative dev turn includes the same expected-failure wrapper rule.
- **AT-VPS-003**: Governed todo example docs and quickstart explicitly recommend a zero-exit verification command (`npm test`, `node --test`, or equivalent) and forbid raw non-zero negative-case commands on a passing turn.
- **AT-VPS-004**: Verification validator error text for `pass` + non-zero exit codes explains the remediation path, not just the failure symptom.

## Open Questions

- None for this slice. This is a guidance and failure-message hardening change, not a protocol or schema change.
