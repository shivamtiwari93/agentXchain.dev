# Repeatable Model-Backed Remote Agent Proof Spec

> Spec for proving that the governed turn-result contract is reliably satisfiable by a real model across repeated runs, not just a single lucky attempt.

## Purpose

The single-run model-backed proof (`run-model-proof.mjs`) demonstrated that Claude Haiku can produce protocol-compliant turn results. But a single pass says nothing about reliability. This spec defines a **repeatable proof harness** that:

1. Runs N independent governed lifecycles through the model-backed bridge
2. Records per-run pass/fail with exact failure mode
3. Reports an honest aggregate pass rate
4. Never retries silently — each run is one attempt, one verdict

If the pass rate is high (>=80% over N>=5 runs), this is real evidence of contract teachability. If it is low, we record the honest failure modes instead of hiding them.

## Scope

### In Scope
- Repeatable proof script that runs the dev-proposed + qa-review lifecycle N times (configurable, default 5)
- Per-run isolation: fresh temp directory, fresh scaffold, fresh turn IDs
- Per-run honest verdict: PASS or FAIL with exact failure reason
- Aggregate report: pass count, fail count, pass rate, per-run breakdown, failure mode taxonomy
- Structured report output (`REPEATED_PROOF_REPORT.md`) with machine-readable summary
- Cost tracking: total tokens and estimated USD across all runs

### Out of Scope
- Multi-model proof (e.g., OpenAI, DeepSeek) — deferred until single-model reliability is proven
- Retry loops or prompt repair between attempts — that would be dishonest
- Parallel execution — runs are serial to avoid port/scaffold conflicts and to produce clear per-run logs
- Performance benchmarking — this is a reliability surface, not a latency surface
- Structured output / tool-use mode — v1 uses free-form text response with the system prompt as the only contract

## Architecture

```
run-repeated-proof.mjs --runs N
  for i in 1..N:
    create fresh tmpdir
    start model-backed bridge on random port
    scaffold governed project with remote_agent runtimes
    agentxchain step --role dev → Claude produces proposed changes
    verify proposal materialization
    agentxchain proposal apply
    agentxchain step --role qa → Claude produces review with objection
    verify review artifact
    record: PASS or FAIL with reason
    cleanup tmpdir
  aggregate results
  write REPEATED_PROOF_REPORT.md
```

## Proof Contract

Each run is independent. The bridge server is reused across runs (same process, same system prompt). Each run gets:
- Fresh temp directory
- Fresh `agentxchain init --governed` scaffold
- Fresh config pointing at the bridge
- Unique run_id and turn_ids (generated by the CLI)

A run PASSES if and only if:
1. `agentxchain step --role dev` exits 0 (or 1 with validation failure, which is then a FAIL)
2. Proposal files exist under `.agentxchain/proposed/<turn_id>/`
3. `agentxchain proposal apply` succeeds
4. `agentxchain step --role qa` exits 0 (or 1 with validation failure)
5. Review artifact exists under `.agentxchain/reviews/`

A run FAILS if any of the above checks fail. The failure reason is recorded precisely (e.g., "dev turn validation failed: empty proposed_changes", "qa turn validation failed: no objections", "model returned non-JSON", "HTTP 502 from bridge").

## Acceptance Tests

1. `run-repeated-proof.mjs --runs 1` completes and writes a report with 1 run entry
2. `run-repeated-proof.mjs --runs 5` completes and writes a report with 5 run entries
3. Report contains honest pass/fail per run — no silent retries
4. Failed runs record the exact failure mode, not a generic message
5. Report contains token usage and cost estimate
6. Script exits 0 regardless of pass rate (it is a reporting tool, not a gate)
7. Script fails-closed if `ANTHROPIC_API_KEY` is missing

## Cost Estimate

- ~$0.01-0.02 per run (Haiku pricing at $1.00/$5.00 per MTok)
- 5 runs: ~$0.05-0.10
- 10 runs: ~$0.10-0.20

## Open Questions

None. Async polling, multi-model, and structured-output mode are all explicitly out of scope.
