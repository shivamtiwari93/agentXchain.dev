# Run a governed flow without lying to yourself

Governed mode is staged and gate-driven. `agentxchain run` is the primary automated workflow, but the scaffold created by `init --governed -y` is mixed-mode by default:

- `pm` uses `manual-pm`
- `dev` uses `local-dev`
- `qa` uses `api-qa`

That means the default scaffold is **not** immediately runnable with `agentxchain run`. If a role is still bound to the `manual` adapter, use `step` for that path or rebind the role to a non-manual runtime first.

## Before you start

- **Node.js 18.17+** (or 20.5+)
- **git** (governed scaffold expects a repo)
- `npx agentxchain` or `npm install -g agentxchain`
- A working local coding CLI for the default `local-dev` runtime, or a config change to point `local-dev.command` at your installed tool
- An API key for any `api_proxy` review role you plan to use, such as `ANTHROPIC_API_KEY` for the default `api-qa` runtime
- Scaffold intent via `--template` or defaults to `pm → dev → qa`

## What governed mode creates

```bash
npx agentxchain init --governed -y
```

This creates the scaffold:

| Category | Files |
|----------|-------|
| Configuration | `agentxchain.json`, `.agentxchain/state.json`, `.agentxchain/history.jsonl`, `.agentxchain/decision-ledger.jsonl` |
| Turn handoff | `.agentxchain/dispatch/turns/<turn_id>/`, `.agentxchain/staging/<turn_id>/turn-result.json`, `TALK.md` |
| Planning artifacts | `.planning/PM_SIGNOFF.md`, `.planning/ROADMAP.md`, `.planning/acceptance-matrix.md`, `.planning/ship-verdict.md` |
| Role prompts | `.agentxchain/prompts/pm.md`, `.agentxchain/prompts/dev.md`, `.agentxchain/prompts/qa.md`, `.agentxchain/prompts/eng_director.md` |

If you already know the repo shape, initialize with a built-in template such as `api-service`, `cli-tool`, `library`, or `web-app`. The template adds project-type planning artifacts and records scaffold intent in `agentxchain.json`. See [Templates](/docs/templates).

## Automated path: use `agentxchain run`

### 1. Initialize

```bash
npx agentxchain init --governed -y
cd my-agentxchain-project
git init && git add -A && git commit -m "initial governed scaffold"
```

### 2. Make the scaffold runnable

The shipped scaffold defaults to `manual-pm`, so `agentxchain run` will stop immediately unless you rebind the PM role to a non-manual runtime. The smallest honest change is to point `pm` at an `api_proxy` runtime and keep `qa` on `api-qa`.

Minimal example:

```json title="agentxchain.json"
{
  "roles": {
    "pm": {
      "runtime": "api-pm"
    }
  },
  "runtimes": {
    "api-pm": {
      "type": "api_proxy",
      "provider": "anthropic",
      "model": "claude-sonnet-4-6",
      "auth_env": "ANTHROPIC_API_KEY"
    }
  }
}
```

If you use Codex CLI, Aider, or another local tool instead of the scaffolded `claude` command, update `runtimes.local-dev.command` before you run anything. See [Adapters](/docs/adapters) for the runtime contract.

### 3. Run the governed flow

```bash
agentxchain run --auto-approve --max-turns 10
```

Use `--auto-approve` for non-interactive execution and CI. Without it, gate pauses still fail closed outside a TTY.

### 4. Inspect the result

```bash
agentxchain status
```

`run` keeps assigning turns, dispatching to adapters, validating staged results, and handling gates until the run reaches a terminal state:

- `completed`
- `blocked`
- `max_turns_reached`

If you want to preview the execution plan without dispatching, use:

```bash
agentxchain run --dry-run
```

## Manual path: keep the shipped mixed-mode scaffold

If you keep `pm` on `manual-pm`, use `step` for the planning turn and let the orchestrator stop where the human work actually happens.

### 1. Assign the PM turn

```bash
agentxchain step --role pm
```

The orchestrator writes a dispatch bundle to `.agentxchain/dispatch/turns/<turn_id>/` and waits for a staged result at `.agentxchain/staging/<turn_id>/turn-result.json`.

### 2. Create PM artifacts and stage the turn result

Write your planning docs (`PM_SIGNOFF.md`, `ROADMAP.md`), then fill the staged turn result. Copy the exact template from the dispatch bundle instead of inventing fields. The minimum truthful shape still looks like this:

```json
{
  "schema_version": "1.0",
  "run_id": "run_...",
  "turn_id": "turn_...",
  "role": "pm",
  "runtime_id": "manual-pm",
  "status": "completed",
  "summary": "Defined the planning scope...",
  "decisions": [
    {
      "id": "DEC-001",
      "category": "scope",
      "statement": "...",
      "rationale": "..."
    }
  ],
  "objections": [
    {
      "id": "OBJ-001",
      "severity": "medium",
      "against_turn_id": "TODO",
      "statement": "...",
      "status": "raised"
    }
  ],
  "files_changed": [],
  "verification": {
    "status": "skipped",
    "commands": [],
    "evidence_summary": "Review turn.",
    "machine_evidence": []
  },
  "artifact": {
    "type": "review",
    "ref": null
  },
  "proposed_next_role": "human",
  "phase_transition_request": "implementation",
  "run_completion_request": null
}
```

Optional validation before acceptance:

```bash
agentxchain validate --mode turn
```

### 3. Accept and approve the gate

```bash
agentxchain accept-turn
agentxchain approve-transition
```

The phase advances from `planning` to `implementation`.

### 4. Continue with dev and QA

```bash
agentxchain step --role dev
agentxchain step --role qa
agentxchain approve-completion
```

QA must challenge the dev's work. Blind agreement is a protocol violation.

## Verify the protocol surface

If you are evaluating AgentXchain as protocol infrastructure, do not stop at a happy-path walkthrough. Run the conformance kit:

```bash
agentxchain verify protocol --tier 3 --target .
```

This exercises the shipped fixture corpus across the state machine, turn-result validation, trust-hardening surfaces, and multi-repo coordinator layer. Add `--format json` when you want CI-readable output.

## Adapter modes

| Adapter | How it works | Automation |
|---------|-------------|------------|
| `manual` | Human reads dispatch, does work, stages result | Full human control |
| `local_cli` | Orchestrator spawns local agent (Claude Code, Codex, Aider) | Fully automated |
| `mcp` | Orchestrator calls a local MCP stdio server tool that returns a governed turn result | Fully automated |
| `api_proxy` | Orchestrator calls LLM API directly with retry and cost tracking | Fully automated |

## Common mistakes

- **Don't treat `run` as zero-config on the default scaffold.** `pm` defaults to `manual-pm`; rebind that role or use `step`.
- **Don't confuse `run` with `step`.** `run` drives a multi-turn automated flow. `step` runs one governed turn and is the right tool for manual roles.
- **Don't confuse `init` with execution.** `init` creates the scaffold. `run` and `step` execute it.
- **Don't edit `state.json` manually.** The orchestrator owns it.
- **Don't skip gates.** `approve-transition` and `approve-completion` are constitutional — they exist so humans stay in control.
- **Turn results must have `objections`.** Blind agreement is a protocol violation.
