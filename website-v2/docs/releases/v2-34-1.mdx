# AgentXchain v2.34.1

`2.34.1` is the releasable cross-session continuity release.

`agentxchain restart` was the right product slice, but the first release prep was not actually shippable. Preflight caught stale version drift in the public docs before anything was pushed, so `2.34.1` is the corrected public release instead of pretending the broken prep never happened.

The underlying product delta remains the same, and it still closes the right long-horizon gap:

- the checkpoint serializer dropped `last_phase` after `approve-transition`
- the spec and CLI proof overstated what was actually guaranteed
- the public release surface still had stale version drift

This release closes those gaps and ships the command on npm with the proof, security, and release-surface work that should have accompanied it from the start.

## Restart checkpoints now reflect the real governed state shape

`session.json` now records checkpoint identity from the same state fields the governed runtime actually persists:

- `last_phase` uses `state.phase` when `current_phase` is absent
- `last_turn_id` recognizes both `id` and `turn_id`
- `last_role` recognizes both review-style and assigned-role turn shapes

That matters because a recovery command is only as truthful as its checkpoint manifest. A null phase in the checkpoint is not “good enough” when the whole point of the feature is long-horizon continuity.

## Restart now has dedicated CLI proof for the missing recovery cases

The restart surface now has subprocess proof for the two cases that were previously under-covered:

- reconnecting to an abandoned active turn in a fresh process
- proving that `accept-turn` and `approve-transition` both update `session.json` through the public CLI

Those tests found the real `last_phase` bug above. This was not speculative hardening. The new proof caught an implementation defect immediately.

## Stale checkpoint mismatch behavior is explicit instead of drifted

The spec and tests now agree with the shipped behavior:

- `state.json` remains the source of truth
- a stale `session.json` `run_id` mismatch produces a warning
- restart continues using the governed state on disk instead of rejecting a recoverable run

Rejecting every stale checkpoint would have been lazy fail-closed theater. Recoverable drift should warn and proceed when the durable governed state is still trustworthy.

## VS Code extension dependency advisories are remediated

The vendored VS Code extension dependency set now pulls patched transitive versions:

- `undici` `6.24.0`
- `brace-expansion` `1.1.13`

That closes the real open advisory surface in `cli/vscode-extension/package-lock.json` instead of pretending the root CLI and website audits were the whole repo.
